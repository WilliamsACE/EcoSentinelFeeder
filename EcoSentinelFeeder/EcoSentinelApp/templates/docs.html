Visión general del sistema · v1.0

Este panel es la extensión web del dispositivo EcoSentinel Feeder — el comedero físico inteligente instalado en la ciudad. La pantalla de inicio muestra el estado global de la red en 4 tarjetas KPI:

🐾 Animales detectados hoyVisitas de perros y gatos en el día.
⚖️ Comida dispensadaKilogramos totales repartidos en todos los feeders.
📡 Feeders en líneaComederos activos y reportando.
⚠️ Alertas activasFeeders con batería baja, comida agotada o sin conexión.

Debajo encontrarás un mapa mini y una tabla con los últimos 5 eventos. Haz clic en cualquier fila para ver el detalle completo.

Arquitectura del sistema físico
🔬
Visión general del prototipo
Raspberry Pi · IA · Dispensador

El prototipo físico es la unidad de campo del sistema EcoSentinel. Corre íntegramente en una Raspberry Pi e integra un sensor PIR, una cámara, detección de movimiento por MOG2, clasificación de especie con YOLO y un dispensador de comida. La arquitectura combina dos procesos separados (proceso principal + YOLO worker) con un hilo de API local para configuración en tiempo real.

⚙️

Todos los parámetros del sistema (confianza, cooldown, fotos por evento, etc.) se gestionan a través de config.py con acceso thread-safe. La FastAPI local en el puerto 8000 permite modificarlos sin reiniciar el sistema.

🗺️proto-arquitectura.pngDiagrama de arquitectura de procesos

Arquitectura de procesos — main_pi.py · detector.py · yolo_worker.py · dispenser.py

CaracterísticaDetalle
🔀 Dos procesos separadosEl proceso principal (con su hilo FastAPI) y el YOLO worker se comunican exclusivamente por una multiprocessing.Queue.
⏱️ Cooldown configurablePor defecto 5 s entre eventos, controlado en el loop principal para evitar procesamiento redundante.
📸 Almacenamiento de fotosLos eventos confirmados se guardan en events_photos/. Las imágenes temporales se eliminan tras la inferencia.
⚙️ config.py centralizadoFuente única de verdad para todos los parámetros del sistema, accesible con bloqueo thread-safe desde cualquier proceso.
Hardware del sistema
🖥️
Componentes de hardware
Raspberry Pi · Sensores · Actuadores

El prototipo se construye alrededor de una Raspberry Pi 4 como unidad de cómputo central. La cámara corre a 320×240 px con backend V4L2 (Linux) para reducir la carga de CPU.

🖥️
Raspberry Pi 4
Unidad central. Ejecuta el proceso principal, el YOLO worker y el hilo FastAPI simultáneamente.
main_pi.py · Process + Thread
📷
Módulo de cámara
320×240 px @30 fps vía V4L2. Captura 4 fotos por evento, vaciando el buffer con 3 cap.grab() previos a cada captura.
OpenCV · CAP_V4L2 o CAP_ANY
🔴
Sensor PIR
Conectado al pin GPIO 17 (BCM). El loop principal lo lee cada 50 ms. Al detectar HIGH se lanza el pipeline.
GPIO 17 · POLL_INTERVAL 50 ms
⚙️
Dispensador
Se activa desde requestDispenser(label) cuando el YOLO worker confirma la especie. Controla el servo y la cantidad de comida a dispensar.
dispenser.py · requestDispenser(label)
Configuración del sistema
⚙️
Configuración dinámica — config.py
Thread-safe · Mutable en tiempo de ejecución

config.py centraliza todos los parámetros operativos del sistema en un diccionario protegido con threading.Lock. Tanto el proceso principal como el YOLO worker y la API local pueden leer y escribir sin race conditions.

ParámetroValor por defectoDescripción
min_confidence0.5Umbral mínimo de confianza YOLO para considerar una detección válida.
min_confirmations2Número mínimo de fotos que deben confirmar la especie para activar el dispensador.
min_contour_area3000Área mínima de contorno MOG2 en px² para considerar que hay movimiento real.
cooldown5.0 sTiempo de espera entre eventos, evita procesamiento redundante del mismo animal.
capture_interval0.5 sIntervalo entre capturas de fotos dentro de un mismo evento.
total_photos4Número de fotos tomadas por evento para enviar al YOLO worker.
debugFalseMuestra la vista en vivo de la cámara (requiere pantalla conectada a la Pi).
Python · config.py
import threading

_lock = threading.Lock()
_cfg  = {
    "min_confidence":    0.5,
    "min_confirmations": 2,
    "min_contour_area":  3000,
    "cooldown":          5.0,
    "capture_interval":  0.5,
    "total_photos":      4,
    "debug":             False,
}

def get(key):
    with _lock: return _cfg[key]

def update(values):
    with _lock:
        for k, v in values.items():
            if k in _cfg: _cfg[k] = v
Flujo de software
🔁
Loop principal — main_pi.py
PIR · Cooldown · Debug mode · event_id

El proceso main_pi.py es el punto de entrada. Al arrancar lanza el hilo FastAPI (daemon) y el proceso YOLO worker, abre la cámara y entra en el loop principal que poll-ea el sensor PIR cada 50 ms.

1
Iniciar FastAPI como hilo daemon
uvicorn.run(fastapi_app, host="0.0.0.0", port=8000) en un Thread(daemon=True). Disponible en /docs.
2
Iniciar YOLO Worker como proceso separado
Process(target=yolo_worker, args=(input_q,), daemon=True).start(). Comparte la Queue con el proceso principal.
3
Abrir cámara (V4L2 → auto)
Prueba CAP_V4L2 y luego CAP_ANY. Fija resolución a 320×240 @30 fps para aliviar la CPU de la Pi.
4
Loop: modo debug
Si config.get("debug") es True, muestra la cámara en vivo con cv2.imshow. Si no hay display disponible, desactiva debug automáticamente y continúa.
5
Loop: verificar cooldown
Si cooldown - (now - last_event_time) > 0 → sleep 50 ms y continuar. Evita eventos repetidos del mismo animal.
6
Loop: leer PIR
Si GPIO.input(PIR_PIN) != HIGH → sleep 50 ms y continuar.
7
Sensor activo → llamar detect()
Registra event_start = time.time(), llama detect(cap, input_q, event_id, event_start). Si retorna True incrementa event_id.
Python · main_pi.py (loop principal)
# Al iniciar:
api_thread = threading.Thread(
    target=lambda: uvicorn.run(fastapi_app, host="0.0.0.0", port=8000),
    daemon=True
)
api_thread.start()

input_q = Queue()
worker  = Process(target=yolo_worker, args=(input_q,), daemon=True)
worker.start()

event_id        = 0
last_event_time = 0.0

while True:
    now = time.time()

    if config.get("debug"):
        # mostrar cámara en vivo; skip resto del loop
        ...
        continue

    if config.get("cooldown") - (now - last_event_time) > 0:
        time.sleep(POLL_INTERVAL)
        continue

    if not sensor_triggered():   # GPIO.input(PIR_PIN) == HIGH
        time.sleep(POLL_INTERVAL)
        continue

    event_start     = time.time()
    last_event_time = event_start
    triggered = detect(cap, input_q, event_id, event_start)
    if triggered:
        event_id += 1
👁️
Detector de movimiento — detector.py
MOG2 · Contornos · Captura con flush de buffer

El módulo detector.py mantiene un sustractor de fondo MOG2 global (history=500, varThreshold=50) que persiste entre llamadas. La función detect() analiza un frame, filtra contornos y si hay movimiento real captura las fotos para el YOLO worker.

💡

Antes de cada captura se ejecutan 3 cap.grab() para vaciar el buffer de la cámara y obtener el frame más reciente. grab() es mucho más rápido que read() porque no decodifica el frame.

1
Leer frame y aplicar MOG2
cap.read() → _back_sub.apply(frame) → máscara de primer plano fg_mask.
2
Encontrar contornos
cv2.findContours(fg_mask, RETR_EXTERNAL, CHAIN_APPROX_SIMPLE). Calcula el área de cada contorno.
3
Filtrar por área mínima
Solo los contornos con área > config.get("min_contour_area") (3000 px²) son significativos. Sin ellos → return False.
4
Capturar total_photos fotos
Por cada foto: 3× cap.grab() + cap.retrieve(), guarda en tempfile.mktemp(".jpg"), pone la tupla en la Queue. Espera capture_interval entre fotos.
5
Retornar True
El proceso principal registra el evento como válido e incrementa event_id.
🤖
YOLO Worker — yolo_worker.py
YOLOv5n · Doble pasada · Early exit · Confirmaciones

El yolo_worker.py corre como proceso independiente y consume la Queue indefinidamente. Carga YOLOv5n.pt una sola vez al arrancar. Para cada foto realiza dos inferencias: una RAW (sin filtro, para logging) y una filtrada (con umbral de confianza y clases objetivo). Mantiene contadores por event_id para implementar la lógica de confirmaciones y early exit.

⚡

Early exit: si el número de confirmaciones alcanza min_confirmations (2) antes de procesar las 4 fotos, llama a requestDispenser() inmediatamente y marca el evento como dispatched para no repetir la dispensación.

1
Cargar modelo (una sola vez)
model = YOLO("yolov5n.pt"). Clases objetivo: [0=person, 15=cat, 16=dog].
2
Pasada RAW — todas las clases
model(img_path, verbose=False) sin filtro de confianza ni clases. Solo para logging de todas las detecciones presentes en el frame.
3
Pasada filtrada — clases objetivo
model(img_path, conf=min_confidence, classes=[0,15,16]). Si detecta: incrementa confirmations[event_id]. Guarda la primera foto confirmada en events_photos/.
4
Eliminar imagen temporal
os.remove(img_path) en el bloque finally. Ocurre siempre, independientemente del resultado.
5
Early exit: ¿confirmaciones ≥ min_confirmations?
Si confirmations[event_id] >= 2 y aún no se ha dispensado → requestDispenser(label), marcar dispatched=True. No espera las fotos restantes.
6
Evento completo: limpiar estado
Cuando received == total_photos: limpia todos los dicts del evento (confirmations, dispatched, received, saved_photo, event_starts, last_label).
Python · yolo_worker.py (lógica principal)
# Dicts por event_id (defaultdict)
confirmations = defaultdict(int)
dispatched    = defaultdict(bool)
received      = defaultdict(int)
last_label    = defaultdict(str)
saved_photo   = defaultdict(bool)
event_starts  = {}

while True:
    event_id, photo_index, total_photos, img_path, event_start = input_q.get()

    # Pasada RAW (logging)
    results_all = model(img_path, verbose=False)

    # Pasada filtrada
    min_conf = config.get("min_confidence")
    results  = model(img_path, conf=min_conf,
                     classes=[0,15,16], verbose=False)
    detected = any(len(r.boxes) > 0 for r in results)

    if detected:
        confirmations[event_id] += 1
        if not saved_photo[event_id]:
            shutil.copy(img_path, EVENTS_SAVE_DIR)
            saved_photo[event_id] = True

    os.remove(img_path)      # siempre en finally
    received[event_id] += 1

    # Early exit
    min_conf_n = config.get("min_confirmations")
    if not dispatched[event_id] and confirmations[event_id] >= min_conf_n:
        requestDispenser(last_label[event_id])
        dispatched[event_id] = True

    # Limpiar al completar evento
    if received[event_id] == total_photos:
        del confirmations[event_id], dispatched[event_id], ...

API local del dispositivo
🔗
API local — FastAPI en puerto 8000
uvicorn · Configuración en tiempo real · /docs

La Raspberry Pi expone una API REST local gestionada por FastAPI y servida con uvicorn en el puerto 8000. Corre como hilo daemon dentro del proceso principal, por lo que no bloquea el loop del sensor. Su función principal es permitir leer y modificar los parámetros de config.py sin reiniciar el sistema.

📖

La documentación interactiva de la API está disponible en http://<ip-raspberry>:8000/docs mientras el dispositivo esté encendido.

GET /config Leer configuración actual

Devuelve un snapshot thread-safe del diccionario _cfg completo con todos los parámetros del sistema en su valor actual.

POST /config Actualizar parámetros

Acepta un JSON con uno o más parámetros a modificar. Los cambios surten efecto de inmediato en el siguiente ciclo del loop sin reiniciar ningún proceso.

JSON — Ejemplo de actualización
{
  "min_confidence":    0.6,
  "min_confirmations": 3,
  "cooldown":          10.0
}
💡

Puedes ajustar debug: true vía API para activar la visualización de la cámara en vivo sin tocar el código. Si la Pi no tiene display conectado, el sistema desactiva el modo debug automáticamente.
